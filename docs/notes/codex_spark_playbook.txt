1) REQUIREMENTS (EXTRACTED FROM FULL CONVERSATION) — DO NOT DROP ANY
-------------------------------------------------------------------------------
User-visible goals (non-negotiable):
  [R1] Python-like syntax (close to Python; MATLAB-like ease for math).
  [R2] Multiparadigm: OOP + Functional + Data-Oriented + Event-Driven built-in.
  [R3] Classes can accept variable/optional inputs like Python (dynamic-ish).
  [R4] First-class math: matrices/lists are central; extremely fast numeric/data workloads.
  [R5] Built-in float types: f8, f16, f32, f64, f128, f256, f512 (and optionally bf16).
  [R6] Primitive baseline: Int (at least i64) and Double (f64) exist and are fast.
  [R7] A single “List” primitive can hold primitives + objects; supports homo/hetero.
  [R8] Object create/access/replace/delete/copy should be near-zero cost when feasible.
  [R9] Language is natively parallel + async-friendly (low overhead concurrency).
  [R10] Memory management must be “innovative” and a differentiator.
  [R11] Runs on (almost) every CPU without serious performance loss.
  [R12] Build time is not important; runtime performance is paramount.
  [R13] Functional pipeline performance must be maximal (fusion, no intermediates).
  [R14] Matrix/list ops: indexing, iteration, replace, delete, matmul, slicing must avoid slowdowns.
  [R15] Heterogeneous lists/matrices must “work”, but fast paths for homogeneous must hit near-max.
  [R16] If a technique risks wrong results, either (a) prove/safeguard it in compile-time layers or
       (b) don’t use it for the “fast/perf tier”.

Engineering constraints (derived):
  [E1] We can create multiple layers/tiers (T0..Tn). Each tier has one main responsibility.
  [E2] Heavy compile-time analyses are allowed: whole-program, auto-tuning, profile-guided, multi-versioning.
  [E3] Target portability must not rely on a slow “one-size-fits-all” runtime; prefer AOT per target + dispatch.

2) REALITY CHECK — “DOABLE” VS “NOT GUARANTEED / PHYSICALLY LIMITED”
-------------------------------------------------------------------------------
You must keep expectations realistic while still pushing aggressively.

Doable:
  - Be 2×–10× faster than *naïve* C for many list/matrix pipelines by eliminating passes/allocations
    (fusion), improving locality, and auto-scheduling (inspired by Weld/stream fusion/Halide/TVM).
  - Reach parity with optimized C for many scalar/array loops (LLVM+LTO+PGO, good layouts).
  - Approach vendor-library performance on some kernels via tiling + auto-tuning; or call vendor libs.

Not guaranteed (treat as “cannot guarantee across all workloads”):
  - “Always 4–5× faster than the best hand-tuned C/ASM for every program.”
    Peak is bounded by memory bandwidth, cache, ISA, and algorithmic limits.
  - “Zero overhead” for fully general heterogeneity with arbitrary per-element operations.
    You can move overhead out of hot loops via analyze+plan+materialize+cache, but not remove it universally.
  - f128/f256/f512 being as fast as f64 on commodity CPUs (most will be software-emulated).
    Provide them for correctness/precision; don’t promise speed.

3) ACADEMIC/THEORETICAL BACKBONE — WHAT WE’RE BUILDING ON (READ + STEAL IDEAS)
-------------------------------------------------------------------------------
You must build the design around proven concepts. Use these as anchors:

(Compilation + IR)
  - MLIR (multi-level IR with extensible dialects; helps heterogenous targets and domain lowering). 
    Ref: https://mlir.llvm.org/
  - Multi-level verification dialects and tooling ideas (robustness for evolving IR stacks).
    Ref: pldi25 “First-Class Verification Dialects for MLIR” (Fehr et al.)

Array/pipeline optimization & schedule separation
  - Halide: separate “algorithm” from “schedule”; scheduling should not change semantics.
    Ref: halide12 paper (Ragan-Kelley et al.), and formal semantics work.
  - Weld: a common IR/runtime that fuses across library boundaries; big gains from reduced data movement.
    Ref: CIDR’17 Weld paper (Palkar et al.), and follow-up works.
  - Stream Fusion: deforestation for list pipelines; eliminate intermediate lists.
    Ref: Coutts/Leshchinskiy/Stewart (2007), and follow-ups.

Functional array languages / performance portability
  - Futhark: purely functional data-parallel arrays; race-free semantics; compiler transforms for performance.
    Ref: PLDI’17 Futhark paper + thesis/book.

Kernel / tiling compilers (matrix-heavy)
  - Triton: tiled IR/compiler for matrix/tensor kernels; near vendor performance in some cases.
    Ref: Tillet et al. (2019).
  - TVM: end-to-end optimizing compiler; graph+operator fusion; auto-tuning/cost models.
    Ref: Chen et al. (OSDI’18 / arXiv).

Dynamic objects at speed (object model)
  - V8 hidden classes (“Maps”) + inline caching: optimize property access by stabilizing shapes.
    Ref: https://v8.dev/docs/hidden-classes and related shape/IC explanations.

Memory management theory
  - Region-based memory management (Tofte & Talpin, 1997).
  - Cyclone regions (Grossman et al., 2002) and modern region/effect variants (e.g., Spegion 2025).

Parallelism + scheduling
  - Cilk work-stealing scheduler: efficient for well-structured parallelism; provable bounds.
    Ref: Blumofe/Leiserson (1996), and later work.
  - Structured concurrency (Swift, Kotlin): lifecycle-scoped tasks, cancellation, safety by structure.
    Refs: Swift concurrency docs, Kotlin coroutine docs.

Profile-guided + post-link optimization (build-time heavy is OK)
  - AutoFDO (sample-based profiling -> FDO), Propeller (profile-guided relinking), BOLT (post-link optimizer).
    These yield single-digit to sometimes double-digit % improvements beyond PGO/LTO in practice.

4) CORE DESIGN DECISION (RECOMMENDED): “PYTHONIC SURFACE, STATIC PERF CORE”
-------------------------------------------------------------------------------
To reconcile Python feel with C-class performance, use a tiered design:

T8: Surface language (Pythonic)
  - Dynamic-feel syntax, flexible classes, convenience features.
  - Allowed: hetero lists, open objects, dynamic calls.
  - Promise: “it runs” + good defaults.

T6: Adaptive runtime tier
  - Performs analyze+plan steps for container layouts (homogeneous fast paths).
  - Inserts guards once; never per-element checks in hot loops if avoidable.

T4: Performance tier (“Perf Core”)
  - Hot code must be stabilizable: known types/shapes/contiguity/effects.
  - Disallows or strongly constrains features that prevent specialization.
  - Goal: generate predictable, low-level code comparable to C/Fortran.

T2: Kernel tier
  - Very small “unsafe-ish” subset used by the compiler to express kernels.
  - Any UB risk is contained here, never exposed to surface.

T0: FFI/Intrinsics (optional)
  - Optional for truly last-mile tuning (intrinsics, asm), but keep minimal.

Key idea:
  - Users write T8.
  - Compiler + analyzer lowers to T4/T2 for hot regions automatically.
  - When impossible, it falls back to slower paths *explicitly* and transparently (with diagnostics).

5) LIST + MATRIX: MULTI-REPRESENTATION WITH STATE MACHINE (THE PERFORMANCE CENTERPIECE)
-------------------------------------------------------------------------------
Design List/Matrix as “one semantic type, many physical reps.”

List semantic API:
  - indexing: x[i]
  - iteration: for v in x
  - update: x[i] = ...
  - append/pop/insert/delete/slice
  - map/filter/reduce/zip/scan (must fuse when possible)

Physical reps (choose dynamically or statically):
  (L1) Packed[T]          : unboxed, contiguous, homogeneous
  (L2) PackedNumeric[U]   : promoted numeric common type (e.g., i64->f64)
  (L3) ChunkedUnion       : segments per type (good for iter-heavy hetero)
  (L4) Boxed[Any]         : tagged cells / pointers (general fallback)
  (L5) GatherScatterView  : typed views + index map (preserve order; fast kernels)
  (L6) SmallList          : inline small-vector optimization (avoid heap for tiny lists)

State machine transitions (core rule):
  - Never pay per-element tag checks inside hot loops if a one-time analyze+materialize can avoid it.
  - “Analyze once, cache plan”:
      layout = analyze(x)
      plan   = choose(layout, op)
      x_rep  = materialize_if_needed(x, plan)
      run fast kernel over x_rep
      cache(x.layout_state = x_rep)

Matrix reps:
  (M1) Matrix[T] contiguous (row-major or column-major; pick one canonical; allow views/strides)
  (M2) MatrixView[T] (slice/transpose) with strides; lower to contiguous when needed
  (M3) BlockedMatrix[T] (tiling-friendly internal layout)
  (M4) Sparse variants (future; optional, inspired by Taichi sparse structures)

Rule for max performance:
  - Matmul/conv/BLAS-like ops should operate on contiguous/blocked typed reps.
  - If surface code creates hetero “matrix”, enforce an explicit normalize (or error in perf tier).

-------------------------------------------------------------------------------
6) OBJECT MODEL: “VALUE/SHAPE FIRST” FOR NEAR-ZERO COST
-------------------------------------------------------------------------------
Goal: make object creation and field access cheap by default.

Three object modes (one semantic class system, multiple runtime reps):
  (O1) Value Object (inlined / unboxed)
      - stored directly in arrays/stack when it doesn’t escape.
      - best for data-oriented and numeric-heavy code.
  (O2) Handle Object (stable id/handle, movable storage)
      - good locality; compaction possible; 1 indirection.
  (O3) Open Object (dict/shape-changing)
      - maximum flexibility; slow path.

Shape system (borrow from V8 “Maps”):
  - Each object carries a ShapeID.
  - Field accesses compile to: guard(shape==X) then load at known offset.
  - If megamorphic site: fallback to dictionary lookup or polymorphic inline cache.

Key optimization gates:
  - Escape analysis decides Value vs Handle vs Open.
  - “Perf tier” restricts frequent shape changes; encourages sealed/slot-like objects.

-------------------------------------------------------------------------------
7) MEMORY MANAGEMENT: “HYBRID REGIONS + GENERATIONAL + ARENAS”
-------------------------------------------------------------------------------
You must differentiate on memory; the user demanded innovation.

Recommended hybrid strategy:
  - Fast arenas / bump allocators for short-lived allocations (expression temps, pipeline intermediates).
  - Region inference or effect-based regions for large scopes (Tofte/Talpin lineage; Cyclone/Spegion ideas).
  - Generational GC OR deferred refcounting ONLY for objects that truly need tracing.
  - Pool allocators for uniform small objects.
  - Copy/move compaction for handle objects to improve locality (optional; advanced).

How this supports performance:
  - Pipeline fusion reduces allocations; remaining allocations go to bump/region -> near-zero cost.
  - Escape analysis + regionization means most temps never hit general heap.
  - Hetero containers can normalize into packed buffers allocated in arenas.

Correctness gates:
  - Debug builds run sanitizers (ASan/UBSan/TSan where applicable).
  - Perf tier forbids undefined behaviors; unsafe ops are isolated in T2 kernel tier.

-------------------------------------------------------------------------------
8) PARALLEL + ASYNC + EVENT-DRIVEN: “STRUCTURED + WORK-STEALING”
-------------------------------------------------------------------------------
Requirements: built-in parallelism and async that doesn’t kill performance.

Model:
  - Structured concurrency for async tasks (Swift/Kotlin style):
      * tasks live inside scopes
      * cancellation propagates
      * clear lifetimes
  - Work-stealing scheduler for fine-grained parallel tasks (Cilk style).
  - Async/await compiles to state machines; avoid allocations for state machines when possible.
  - Event loop primitives (channels, streams, async iterators).

Safety/perf tier:
  - “Sendable”-like trait/effect: what can cross threads.
  - Ownership/uniqueness/effect system to prevent data races in perf-tier code.
  - Zero-cost when used sequentially (no forced runtime checks in hot loops).

-------------------------------------------------------------------------------
9) PORTABILITY WITHOUT BIG PERF LOSS
-------------------------------------------------------------------------------
Approach:
  - Use LLVM/MLIR to generate native code for multiple architectures.
  - Produce “fat binaries”:
      * baseline (portable ISA subset)
      * tuned variants (AVX2/AVX-512 on x86; NEON/SVE on arm; etc.)
      * runtime dispatch based on CPUID/HWCAP.
  - For “runs everywhere” fallback, optionally support WebAssembly as a *compat mode*,
    but do NOT claim it matches native peak everywhere.

-------------------------------------------------------------------------------
10) PERFORMANCE TOOLCHAIN: USE BUILD-TIME HEAVY OPTIONS
-------------------------------------------------------------------------------
Because build time is free, use:
  - LTO/ThinLTO
  - PGO (instrumentation or sampling)
  - Post-link optimization (BOLT)
  - Profile-guided relinking (Propeller-like workflow)
  - Sample-based FDO (AutoFDO concepts)
This can yield single-digit % improvements on top of already-optimized builds (sometimes more).

PHASE 3 — Type/Shape system + Tier boundaries (T8->T4 lowering)
Goal:
  - Make “fast subset” explicit and machine-checkable.
Tasks:
  1) Implement type system:
       - Int (i64), Double (f64), Float<N> family, Bool, String (optional),

  - Shape stabilization tests (same fields -> same shape).

  - Compiler can label regions as “Perf-tier eligible” or “needs fallback” with clear reasons.

PHASE 4 — Codegen v0 (scalar): LLVM backend + runtime launcher
Goal:
  - Generate native code for scalar code paths.
Tasks:
  1) Choose IR strategy:
       - Option A: Lower to MLIR first, then to LLVM dialect, then LLVM IR.
       - Option B: Direct LLVM IR from your own IR (faster to start, less future-proof).
     Recommendation: Option A if you commit to MLIR stack; Option B only for prototyping.
  2) Implement codegen for ints/doubles, control flow, function calls.
  3) Implement runtime launcher (JIT or AOT; AOT preferred for reproducibility).
Verification checklist:
  - Run scalar benchmarks and compare vs clang -O3.
Exit criteria:
  - On scalar loops, within 0.9×–1.2× of clang -O3 (same algorithm).

PHASE 5 — Containers v1: Packed[T] list + Matrix[T] + bounds proof hooks
Goal:
  - “Homogeneous fast path” must be excellent.
Tasks:
  1) Implement Packed[T] list:
       - contiguous, unboxed, aligned
       - amortized append strategy
  2) Implement Matrix[T]:
       - contiguous storage
       - views/slices as stride-based references
  3) Add bounds-check elimination hooks:
       - range analysis in loops
       - hoist checks or prove safe
Verification checklist:
  - Bench B2/B4 on packed data.
Exit criteria:
  - Packed list iteration/map/reduce is within 0.85×–1.15× of equivalent C loops.
  - Matrix elementwise ops within 0.85×–1.15× of C loops.

PHASE 6 — Containers v2: heterogeneity without killing hot loops
Goal:
  - Hetero lists “work”, but hot loops pay almost nothing after normalization.
Tasks:
  1) Implement Boxed[Any] representation (tagged cells / handles).
  2) Implement analyze+plan+materialize+cache for list ops.
  3) Implement promotion for numeric heterogeneity (PackedNumeric[U]).
  4) Implement ChunkedUnion for iter-heavy hetero patterns.
  5) Implement GatherScatter views for order-preserving typed kernels.
Verification checklist:
  - Bench B3 and show:
       * first run may pay normalize cost
       * subsequent runs are fast (cached rep)
Exit criteria:
  - For repeated operations on the same hetero list, steady-state performance approaches packed case
    (target: within 1.2×–1.8× depending on mix and required semantics).

PHASE 7 — Functional pipelines: fusion IR + no-intermediate execution
Goal:
  - Functional style must be *fast by default*.
Tasks:
  1) Introduce Array/Pipeline IR (Weld/stream-fusion inspired):
       - represent map/filter/reduce/zip/scan at a high level
       - enable fusion and loop transformations
  2) Implement fusion passes:
       - remove intermediates
       - push filters into producers
       - deforest list pipelines where safe
  3) Materialization rules:
       - only materialize when user observes container identity or needs random access
Verification checklist:
  - Bench B5: pipeline should allocate ~0 intermediates (or bounded minimal).
Exit criteria:
  - Pipeline throughput beats non-fused baseline by 2×–10× on large data (when baseline allocates intermediates).
  - Allocation count for pipeline is near-zero or strictly bounded.

PHASE 8 — Matrix kernel engine: tiling + scheduling (+ optional auto-tuning)
Goal:
  - Matmul and core kernels are competitive.
Tasks:
  1) Implement schedule abstraction (Halide-style): algorithm separate from schedule.
  2) Provide default schedules + auto-tuner (TVM-style search) — build-time is allowed to be heavy.
  3) Implement matmul:
       - blocking/tiling, packing, cache-aware layout
       - support MatrixView lowering when needed (copy-on-write/pack)
  4) (Optional) Call BLAS initially, then replace with generated kernels.
Verification checklist:
  - Bench B4 (matmul sizes).
Exit criteria (choose one path):
  Path A (own kernels): achieve within ~1.2×–2.0× of OpenBLAS on common sizes initially, improve over time.
  Path B (hybrid): match BLAS for matmul by calling it, while your compiler still fuses surrounding ops to reduce overhead.

PHASE 9 — Concurrency: structured async + work stealing + safety rules
Goal:
  - Async and parallelism are first-class, low overhead, safe in perf tier.
Tasks:
  1) Implement async/await lowering to state machines.
  2) Implement structured concurrency scopes and cancellation.
  3) Implement a work-stealing scheduler for parallel tasks.
  4) Implement Send/Sync-like rules via effects/ownership for perf tier.
Verification checklist:
  - Bench B7/B8 for overhead and scaling.
Exit criteria:
  - Task spawn/join overhead is competitive (measure and record).
  - Parallel map scales reasonably on multicore (document scaling curve).

PHASE 10 — Portability + “final-mile” optimization + hardening
Goal:
  - Ship multi-arch builds and squeeze remaining performance.
Tasks:
  1) Multi-target builds (x86_64, aarch64, riscv64) + baseline ISA profile.
  2) Multiversion dispatch for CPU features (build multiple variants).
  3) PGO + LTO + post-link BOLT integration.

Exit criteria:

13) COMPARISON: WHAT TO “STEAL” FROM POPULAR LANGUAGES (ACTIONABLE)
-------------------------------------------------------------------------------
Rust:
  - Ownership/borrowing to reduce aliasing and enable safe concurrency without GC.
  - Takeaway: use compile-time rules/effects to unlock optimization (bounds/alias elimination).
Zig:
  - Explicit allocator passing + compile-time evaluation (“comptime”) for specialization.
  - Takeaway: make allocation strategy explicit in perf tier; use compile-time specialization.
Julia:
  - Specialization to reconcile dynamism and performance; multiple dispatch as a performance tool.
  - Takeaway: specialization is powerful, but hetero containers cause dynamic dispatch slowdowns; mitigate via normalization.
Chapel:
  - Parallelism as a first-class language feature for productivity at scale.
  - Takeaway: provide high-level parallel constructs that lower to efficient runtimes.

Swift/Kotlin:
  - Structured concurrency; clear task lifetimes; cancellation semantics.
  - Takeaway: integrate structured concurrency to keep async correct and predictable.

  - pending-task sinyali ile cv-wake modeli eklendi.

  - parallel_for/par_map/par_reduce icinde per-iteration arg vector allocation azaltildi.

    (paylasilan mutable parent env contention/race riski azaltildi).
- Benchmark duzeltmeleri:
  - parallel_for scaling programi chunk-lock contention'dan arindirildi
    (gercek scheduler scaling gorunur hale getirildi).
